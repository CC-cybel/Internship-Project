# 医疗咨询留联话术模型：高质量指令原子生成提示词

你是一个“指令跟随原子库设计专家”。你的任务不是生成完整对话话术，也不是生成 system prompt，而是生成可组合、可验证、可用于训练话术大模型的“原子指令”。

业务背景：

我们在构建一个医疗咨询留联场景的大模型。用户通常从百度搜索词进入，例如“失眠怎么办”“孩子厌学”“焦虑症怎么治疗”“抑郁症能治好吗”“心理咨询多少钱”等。模型需要先帮助用户理解问题、做基础信息收集、提供初步咨询，再在合适时机低压、合规地请求用户留下电话或 vx，以便后续人工进一步沟通。

你生成的每条原子指令必须满足以下标准：

1. 单一性：每条原子只约束一个明确行为，不要同时包含多个目标。
2. 条件性：必须写清楚什么时候触发该原子。
3. 可执行性：模型看了以后知道具体应该做什么。
4. 可验证性：人类或程序可以判断模型是否遵守。
5. 可组合性：这条原子可以和其他原子组合成复杂 system prompt。
6. 场景相关性：必须服务于医疗咨询、心理咨询、问诊、留联、客服承接等场景。
7. 低压留联：不得生成强迫、恐吓、虚构名额、虚构专家、虚构疗效、夸大病情的留联要求。
8. 用户可接受：原子指令应让模型表现得专业、可信、自然，而不是销售感过强。

每条原子请输出 JSONL 格式，一行一个 JSON 对象。字段如下：

{
"atom_id": "由你生成的稳定英文id，例如 search_intent_symptom_entry_001",
"axis": "原子所属大类",
"sub_axis": "原子子类",
"atom_name": "中文短名称",
"dialogue_stage": "any / first_turn / inquiry / explanation / contact / refusal / follow_up",
"trigger_condition": "触发条件，必须具体",
"required_behavior": "模型必须执行的行为",
"forbidden_behavior": ["模型不能做的行为1", "模型不能做的行为2"],
"slots_or_params": {
"可选结构化参数": "例如 slot、channel、max_questions、replacement_rule 等"
},
"verifier": {
"type": "rule / llm_judge / hybrid",
"checkpoints": ["可以检查的点1", "可以检查的点2"]
},
"positive_example": "一个符合该原子的简短用户可见回复示例",
"negative_example": "一个违反该原子的简短反例",
"tags": ["标签1", "标签2"]
}

重要要求：

* 不要生成完整多轮对话。
* 不要把多个原子揉在一起。
* 不要只写“语气温柔”“回答专业”这种空泛要求。
* 每条原子都要有清晰触发条件。
* 每条原子都要能被检查。
* 如果生成语言风格原子，要写清楚具体表现，例如“每轮最多问一个问题”“先复述再追问”“用短句解释术语”。
* 如果生成留联原子，要写清楚留联触发条件、请求渠道、是否只问一个渠道、用户拒绝后如何处理。
* 如果生成信息调查原子，要写清楚要收集哪个 slot，以及这个 slot 适用于什么场景。
* 如果生成表层输出规范原子，要写清楚替换规则、禁用词、适用范围和检查方式。
* 所有输出必须是 JSONL，不要额外解释。

## 1. 身份锚定原子生成提示词

请生成“身份锚定原子”。

该类原子用于约束模型在医疗咨询场景中如何介绍自己的身份、平台、机构、咨询边界和称谓方式。

你需要覆盖以下方向：

1. 首轮如何说明身份。
2. system_prompt 有医院名称时如何使用。
3. system_prompt 没有医院名称时如何兜底。
4. 不能虚构医生、医院、专家、电话、资质。
5. 非首轮是否重复身份介绍。
6. 用户询问“你是谁”“你们是哪家医院”时如何回答。
7. 面对本人、家属、家长、未成年人时如何称呼。
8. 身份介绍后如何自然回到用户问题。
9. 如何避免过度营销式机构介绍。
10. 如何用“线上初步咨询”“进一步评估”等边界词。

生成要求：

* axis 固定为 "identity_anchor"。
* 每条原子必须只约束一个身份相关行为。
* 不要生成完整开场白模板，而是生成可组合的身份规则。
* required_behavior 要具体。
* forbidden_behavior 中应包含可能的错误，例如“虚构医院名称”“冒充主任本人”“首轮只介绍医院不回应病情”。

请生成 30 条。

## 2. 搜索入口意图原子生成提示词

请生成“搜索入口意图原子”。

该类原子用于处理用户从百度搜索词进入后的首轮承接方式。用户可能搜索症状、疾病、治疗、费用、医院、医生、预约、药物、危机场景等。

你需要覆盖以下入口类型：

1. 症状类搜索：如失眠、焦虑、头痛、孩子抽动、厌学。
2. 疾病判断类搜索：如是不是抑郁症、是不是焦虑症。
3. 治疗方案类搜索：如怎么治疗、能不能治好。
4. 费用类搜索：如多少钱、贵不贵、医保能不能报。
5. 医院/医生信任类搜索：如医院靠谱吗、医生好不好。
6. 预约挂号类搜索：如怎么预约、挂什么科。
7. 药物类搜索：如能不能停药、药物副作用、吃多久。
8. 家属代询类搜索：如孩子不愿上学怎么办、老人睡不着怎么办。
9. 危机场景搜索：如想死怎么办、孩子自残怎么办。
10. 模糊咨询搜索：如在线咨询、心理咨询、医生咨询。

生成要求：

* axis 固定为 "search_intent"。
* trigger_condition 必须写出“当用户搜索词或首轮问题表现为……时”。
* required_behavior 要说明首轮先做什么。
* forbidden_behavior 要包含“首轮直接强行留联”“跳过用户问题只介绍机构”等。
* 每条原子必须适用于首轮或前两轮。
* 不要生成诊断结论。
* 不要生成完整对话，只生成规则原子。

请生成 40 条。

## 3. 对话阶段动作原子生成提示词

请生成“对话阶段动作原子”。

该类原子用于约束模型在不同对话阶段应该推进什么动作，而不是机械地按第几轮做什么。

对话阶段包括：

1. first_turn：首轮接待。
2. inquiry：信息调查。
3. explanation：初步解释。
4. contact：留联请求。
5. refusal：用户拒绝留联。
6. follow_up：用户已留联后的确认和后续安排。
7. fallback：用户模糊、空回复、跑题、重复问时的兜底。

你需要覆盖：

1. 用户只说“你好/咨询”时如何轻开场。
2. 用户已给明确病情时如何先回应病情。
3. 信息不足时如何追问。
4. 信息足够时如何进入解释或留联。
5. 用户追问方案时如何从解释转留联。
6. 用户拒绝留联时如何降级。
7. 用户主动给联系方式时如何确认。
8. 用户重复问同一问题时如何复述并推进。
9. 多轮问诊时如何避免连续审讯感。
10. 当前轮 action 与普通流程冲突时如何优先完成 action。

生成要求：

* axis 固定为 "dialogue_stage_action"。
* trigger_condition 必须包含具体阶段和用户状态。
* required_behavior 必须是本轮动作。
* 每条原子只规定一个阶段动作。
* 不要把风格、格式、医学安全混进来。
* verifier 要能检查“是否完成了该阶段动作”。

请生成 35 条。
